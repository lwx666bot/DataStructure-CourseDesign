============================================================
                排队叫号子系统开发设计说明
============================================================

一、整体设计思想
------------------------------------------------------------
本子系统采用“系统类 + 顾客结构体 + 队列容器”的方式实现。

基本思路是：

1. 使用 Customer 结构体保存单个顾客的信息。
   例如：排队号码、顾客姓名、到达时间等。

2. 使用 QueueSystem 类表示整个排队叫号系统。
   该类内部维护一个 queue<Customer> 队列，用来保存正在等待的顾客。

3. QueueSystem 类对外提供各种成员函数。
   例如：
   addCustomer()       顾客取号排队
   callCustomer()      叫号
   showQueue()         查看当前等待队列
   clearQueue()        清空等待队列
   showRecords()       查看历史记录
   saveRecordsToFile() 保存历史记录到文件

4. 主函数 main() 只负责主菜单控制。
   队列系统内部的数据和操作都交给 QueueSystem 对象完成。

------------------------------------------------------------

二、QueueSystem 对象的创建位置
------------------------------------------------------------
QueueSystem 对象建议在 main() 函数中创建一次，而不是每次进入
排队叫号子系统时重新创建。

推荐思路：

在 main() 中创建：

    QueueSystem queueSystem;

然后进入排队叫号子系统时，把这个对象传给子系统函数：

    runQueueSystem(queueSystem);

这样做的好处是：

1. 只要整个程序没有退出，queueSystem 对象就一直存在。
2. 从排队叫号子系统返回主菜单时，队列数据不会丢失。
3. 再次进入排队叫号子系统时，仍然可以看到之前的排队数据。
4. 这更符合真实系统逻辑：返回主菜单只是切换功能，不代表清空队列。

------------------------------------------------------------

三、为什么不要在 runQueueSystem() 中创建 QueueSystem 对象
------------------------------------------------------------
不推荐在 runQueueSystem() 函数内部创建 QueueSystem 对象。

原因是：

1. 如果 QueueSystem 对象写在 runQueueSystem() 内部，它就是局部变量。
2. 每次进入 runQueueSystem()，都会创建一个新的 QueueSystem 对象。
3. 一旦从 runQueueSystem() 返回主菜单，这个对象就会被销毁。
4. 下次再进入排队叫号系统时，之前的队列数据就全部丢失了。

举例：

第一次进入排队系统：
    1号、2号、3号顾客排队

返回主菜单后：
    runQueueSystem() 结束，局部对象被销毁

第二次进入排队系统：
    队列重新变为空

这种效果不符合实际使用需求，所以不推荐。

------------------------------------------------------------

四、runQueueSystem() 应该使用引用传参
------------------------------------------------------------
推荐写法：

    void runQueueSystem(QueueSystem& queueSystem)

这里的 & 表示引用传参。

作用是：

1. runQueueSystem() 操作的是 main() 中创建的那个原始对象。
2. 不会复制新的 QueueSystem 对象。
3. 在子菜单中添加顾客、叫号、清空队列等操作，都会真正作用到原对象上。
4. 返回主菜单后，数据仍然保留。

不推荐写法：

    void runQueueSystem(QueueSystem queueSystem)

原因是：

1. 这种写法是值传递。
2. 会复制一个新的 QueueSystem 对象。
3. 在函数内部操作的是副本，不是 main() 里的原对象。
4. 可能导致数据修改后无法真正保留下来。

------------------------------------------------------------

五、数据是否应该保留
------------------------------------------------------------
本系统的数据保留原则如下：

1. 从排队叫号子系统返回主菜单时，应该保留当前队列数据。

   原因：
   返回主菜单只是切换功能，不代表排队顾客消失。
   例如顾客已经取号排队，即使管理员返回主菜单，
   再次进入排队系统时，也应该还能看到这些顾客。

2. 退出整个程序后，当前等待队列可以不保留。

   原因：
   当前等待队列保存在内存中。
   程序关闭后，内存释放，队列数据自然消失。
   对于课程设计来说，这是可以接受的。

3. 历史服务记录建议保存到文件。

   原因：
   当前正在等待的顾客可以随着程序退出而消失，
   但已经叫号、已经办理、过号等历史记录具有统计价值。
   可以保存到 txt 或 csv 文件中，用于后续查看和统计。

------------------------------------------------------------

六、推荐的数据保存策略
------------------------------------------------------------
建议区分两类数据：

1. 当前等待队列

   存储位置：
       queue<Customer> customerQueue;

   特点：
       只在程序运行期间有效。
       程序退出后可以清空。

   保存内容：
       正在等待叫号的顾客。

2. 历史记录

   存储位置：
       vector<ServiceRecord> records;

   特点：
       可以在程序运行期间查看。
       也可以写入文件长期保存。

   保存内容：
       已叫号顾客编号
       服务窗口编号
       到达时间
       叫号时间
       等待时间
       服务状态，例如“已办理”“过号”

------------------------------------------------------------

七、推荐的基本类结构
------------------------------------------------------------
可以先按照下面的结构开发第一版：

    struct Customer
    {
        int number;          // 排队号码
        string name;         // 顾客姓名
        time_t arriveTime;   // 到达时间
    };

    class QueueSystem
    {
    private:
        queue<Customer> customerQueue;  // 等待队列
        int nextNumber;                 // 下一个排队号码

    public:
        QueueSystem();

        void addCustomer();     // 顾客取号
        void callCustomer();    // 叫号
        void showQueue();       // 查看等待队列
        void clearQueue();      // 清空等待队列
    };

第一版先完成：

    取号 -> 入队 -> 叫号 -> 出队 -> 显示等待时间

等第一版稳定运行后，再扩展：

    多窗口叫号
    过号处理
    历史记录
    文件保存
    平均等待时间统计

------------------------------------------------------------

八、推荐的主菜单调用方式
------------------------------------------------------------
主函数中创建 QueueSystem 对象：

    int main()
    {
        QueueSystem queueSystem;

        while (true)
        {
            printMainMenu();
            int choice = getMenuChoice();

            switch (choice)
            {
            case 1:
                runQueueSystem(queueSystem);
                break;

            case 0:
                cout << "系统已退出。" << endl;
                return 0;

            default:
                cout << "输入错误，请重新选择。" << endl;
                break;
            }
        }
    }

排队叫号子系统函数：

    void runQueueSystem(QueueSystem& queueSystem)
    {
        while (true)
        {
            printQueueMenu();
            int choice = getMenuChoice();

            switch (choice)
            {
            case 1:
                queueSystem.addCustomer();
                break;

            case 2:
                queueSystem.callCustomer();
                break;

            case 3:
                queueSystem.showQueue();
                break;

            case 4:
                queueSystem.clearQueue();
                break;

            case 0:
                return;

            default:
                cout << "输入错误，请重新选择。" << endl;
                break;
            }
        }
    }

注意：

    case 0:
        return;

这里只是退出排队叫号子系统，返回主菜单。
并不会销毁 main() 中的 queueSystem 对象，
所以队列数据仍然保留。

------------------------------------------------------------

九、开发顺序建议
------------------------------------------------------------
建议按照以下顺序开发：

1. 先写 Customer 结构体。
2. 再写 QueueSystem 类的基本框架。
3. 实现 addCustomer()，完成顾客取号入队。
4. 实现 callCustomer()，完成队首顾客叫号出队。
5. 实现 showQueue()，查看当前等待队列。
6. 实现 clearQueue()，手动清空等待队列。
7. 将 QueueSystem 对象放在 main() 中创建。
8. 使用 runQueueSystem(QueueSystem& queueSystem) 进入子系统。
9. 第一版运行稳定后，再添加历史记录和文件保存。
10. 最后再考虑多窗口叫号和过号处理。

------------------------------------------------------------

十、核心原则总结
------------------------------------------------------------
1. main() 负责整体菜单控制。
2. QueueSystem 类负责排队系统的数据和操作。
3. Customer 结构体负责保存单个顾客信息。
4. queue<Customer> 用来保存等待队列。
5. QueueSystem 对象应该在 main() 中创建一次。
6. 子系统函数使用引用传参，避免数据丢失。
7. 返回主菜单时保留队列数据。
8. 退出整个程序后，当前等待队列可以消失。
9. 历史记录可以保存到文件，方便统计。
10. 先完成核心流程，再逐步扩展复杂功能。
